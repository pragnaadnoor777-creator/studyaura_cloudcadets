## The Problem

Students are overwhelmed. Between assignments, exams, lab records, and everything else life throws at them, cognitive overload is the default state — not the exception. Traditional study tools don't understand context. They don't know what you were just studying. They don't adapt. They don't remember you.

StudyAura does.

---

## What is StudyAura?

StudyAura is a full-stack, AI-native study platform that uses a serverless AWS backend to give every student a personalized, context-aware learning experience. It captures your mental chaos, organizes it with AI, lets you chat with a study assistant that actually remembers your conversation, and then challenges you with a dynamically generated Knowledge Dungeon — a gamified quiz experience built entirely from what *you* were just studying.

No generic flashcards. No static quizzes. Every dungeon boss is unique to you.

## AWS Architecture

StudyAura is built on a fully serverless AWS stack. Every user interaction that requires intelligence flows through the same pipeline:

```
User Interface (React/Vite)
        │
        ▼
  API Gateway (REST)
        │
        ▼
  AWS Lambda (Node.js)
   ├── Fetch chat history from DynamoDB
   ├── Build context-aware prompt
   └── Invoke Amazon Bedrock
        │
        ▼
  Amazon Bedrock (Claude 3 Sonnet)
        │
        ▼
  AWS Lambda (response handling)
   ├── Parse Bedrock response
   ├── Persist new messages to DynamoDB
   └── Return structured JSON to client
        │
        ▼
  DynamoDB (persistence layer)
  S3 (static assets & media)
```

### Service Responsibilities

**Amazon API Gateway**
The single entry point for all client requests. Routes incoming HTTP POST requests to the appropriate Lambda function based on path (`/analyze`, `/chat`, `/dungeon`). Handles CORS, request validation, and throttling.

**AWS Lambda**
Three dedicated Lambda functions handle all backend logic:

- `analyzeChaos` — Processes Brain Dump submissions. Fetches chat history, invokes Bedrock for mental load analysis, persists results.
- `chatHandler` — Manages the AI chat loop. Retrieves the last 5 messages from DynamoDB, sends them to Bedrock as conversation context, saves the new exchange.
- `dungeonMaster` — The most sophisticated function. Pulls the last 10 messages from `StudyAura-ChatHistory`, passes them directly to Bedrock to generate a Knowledge Dungeon challenge that is semantically tied to what the student was just studying. Also handles answer verification and XP awarding.

**Amazon Bedrock (Claude 3 Sonnet)**
The AI engine for the entire platform. Bedrock handles:
- Mental load classification (`HIGH` / `MODERATE`) and study plan generation
- Context-aware chat responses across three conversation modes
- Dynamic Knowledge Dungeon question generation from chat history
- Semantic answer verification (allows paraphrasing, not just exact string matching)

**Amazon DynamoDB**
Two tables power the persistence layer:

- `StudyAura-Progress` — Stores `userId`, `totalXP`, and `clearedRooms`. Updated atomically via `UpdateItemCommand` when a student answers correctly.
- `StudyAura-ChatHistory` — The critical memory store. Every user message and AI response is written here with a `timestamp` sort key. Lambda queries this table before every Bedrock invocation to provide conversation context.

**Amazon S3**
Hosts static frontend assets and any media files served by the application.

## The Context-Aware AI Loop

This is the architectural detail that makes StudyAura genuinely intelligent rather than just another chatbot wrapper.

The core insight: **a quiz is only useful if it tests what you actually studied.** Generic question banks don't do that. StudyAura solves this by wiring the Knowledge Dungeon directly to the student's own conversation history.

Here is exactly how it works, end to end:

### Step 1 — The Student Studies

The student uses the StudyAura AI Chat to work through a topic. Every exchange — every user message and every AI reply — is written to `StudyAura-ChatHistory` in DynamoDB with a `userId` partition key and a `timestamp` sort key.

```js
// chatHandler.js — persisting every exchange
await Promise.all([
  dynamoClient.send(new PutItemCommand({
    TableName: "StudyAura-ChatHistory",
    Item: {
      userId: { S: userId },
      timestamp: { N: String(timestamp) },
      role: { S: "user" },
      message: { S: userMessage },
    },
  })),
  dynamoClient.send(new PutItemCommand({
    TableName: "StudyAura-ChatHistory",
    Item: {
      userId: { S: userId },
      timestamp: { N: String(timestamp + 1) },
      role: { S: "ai" },
      message: { S: aiReply },
    },
  })),
]);
```

### Step 2 — The Student Enters the Dungeon

When the student clicks "Enter Dungeon", the frontend fires a POST to `/dungeon` with `action: "generate"`. The `dungeonMaster` Lambda is invoked.

### Step 3 — Lambda Fetches the Memory

Before touching Bedrock, Lambda queries DynamoDB for the **last 10 messages** from that student's chat history, sorted descending by timestamp, then reversed to chronological order for the prompt.

```js
// dungeonMaster.js — fetching the student's study context
const result = await dynamoClient.send(
  new QueryCommand({
    TableName: "StudyAura-ChatHistory",
    KeyConditionExpression: "userId = :uid",
    ExpressionAttributeValues: { ":uid": { S: userId } },
    ScanIndexForward: false, // newest first
    Limit: 10,
  })
);

// Reverse to chronological order before passing to Bedrock
return (result.Items || []).reverse().map((item) => ({
  role: item.role.S === "ai" ? "assistant" : "user",
  content: item.message.S,
}));
```

### Step 4 — Bedrock Generates a Personalized Boss Battle

Lambda passes the full chat history directly into the Bedrock `messages` array, followed by a system prompt instructing Claude to generate a dungeon challenge *specifically about the topic the student was just discussing*. Bedrock reads the conversation, understands the subject matter, and returns a structured JSON challenge.

```js
// dungeonMaster.js — the context-aware Bedrock invocation
const body = JSON.stringify({
  anthropic_version: "bedrock-2023-05-31",
  max_tokens: 1024,
  system: "You are StudyAura's Dungeon Master AI. Based on the student's recent study conversation, generate a Knowledge Dungeon challenge.",
  messages: [
    ...chatHistory,  // <-- the student's actual study session
    {
      role: "user",
      content: `Generate a Knowledge Dungeon challenge based specifically on the topic the user was just discussing.
      
Respond with:
{
  "roomName": "creative dungeon room name related to the topic",
  "question": "a clear, specific question about the topic",
  "correctAnswer": "the correct answer",
  "explanation": "brief explanation of why this is correct"
}`
    },
  ],
});
```

### Step 5 — Semantic Answer Verification

When the student submits an answer, Lambda calls Bedrock *again* — this time as a semantic verification engine. Rather than a brittle string comparison, Bedrock evaluates whether the student's answer is *essentially correct*, allowing for natural paraphrasing and minor wording differences. If correct, Lambda fires an `UpdateItemCommand` to atomically add +20 XP to `StudyAura-Progress`.

```js
// dungeonMaster.js — semantic answer checking
{ "isCorrect": true or false, "feedback": "brief encouraging feedback" }
```

### The Result

A student who spent 20 minutes chatting about Big-O notation will enter a dungeon with rooms named things like *"The Complexity Catacombs"* and face questions about time complexity — not random trivia. A student who studied the French Revolution gets questions about the French Revolution. The dungeon is a direct reflection of the student's own learning session, made possible entirely by the DynamoDB memory layer feeding context into every Bedrock invocation.

## DynamoDB Schema

### `StudyAura-ChatHistory`

| Attribute | Type | Description |
|---|---|---|
| `userId` | String (PK) | Unique student identifier |
| `timestamp` | Number (SK) | Unix timestamp in milliseconds |
| `role` | String | `"user"` or `"ai"` |
| `message` | String | The message content |

### `StudyAura-Progress`

| Attribute | Type | Description |
|---|---|---|
| `userId` | String (PK) | Unique student identifier |
| `totalXP` | Number | Cumulative XP earned |
| `clearedRooms` | List | Array of cleared dungeon room IDs |

## Local Development

### Prerequisites

- Node.js 18+
- AWS account with Bedrock access enabled for Claude 3 Sonnet
- DynamoDB tables provisioned (see schema above)
- API Gateway + Lambda deployed

### Frontend Setup

```bash
cd frontend
npm install
```

Create a `.env.local` file:

```env
VITE_AWS_API_URL=https://your-api-gateway-url.execute-api.us-east-1.amazonaws.com/dev
```

```bash
npm run dev
```

### Backend Environment Variables

Set these on your Lambda functions:

```env
AWS_REGION=us-east-1
BEDROCK_MODEL_ID=anthropic.claude-3-sonnet-20240229-v1:0
```
